# Dimensional Programming

## Purpose
Dimensional Programming is Butterfly IDE's kernel doctrine: store rules and coordinates, derive meaning through substrates, and execute effects through operations.

It is not merely modularity in the ordinary software sense. Dimensional Programming means modularity building upon modularity until stable dimensional bands emerge. Modules are seed forms that can accumulate, relate, collapse, and promote into higher-order wholes.

## Core Law
Everything is represented as one or more of the following:
- coordinates
- relations
- derived expressions
- operations

Nothing should be stored twice if it can be derived once.

## Dimensional Construction Law
Butterfly does not treat a module as the finished unit of design. A module is a point of creation inside a larger dimensional ladder.

The doctrine is therefore:
- build clear modules
- let modules accumulate into structures
- let structures accumulate into dimensional bands
- let mature bands collapse into promoted points for the next dimension

So modularity is only the beginning. Dimensional Programming is the disciplined growth of modular units into higher-order coordinated form.

## Governing Geometry
The primary manifold is:

`z = x * y`

This manifold is the geometric source of the system. It is not a payload store. It is the shared surface from which substrates measure, classify, and derive usable results.

## Hardened Surface Principle
Butterfly IDE is governed by the minimal-surface intuition of related families such as the Schwarz Diamond and the gyroid.

This yields three system laws:

### Minimal material
- store the fewest durable rules necessary
- prefer derivation over duplication
- reduce architectural mass before adding new structure

### Maximal area
- one manifold should support many substrates
- one substrate should support many projections
- one explanation should generalize across many cases

### Maximal strength
- pivots must be typed and explicit
- projection and collapse must be deterministic
- execution must remain policy-bounded and auditable

## Primary Helix Manifold
The hardened runtime interprets `z = x * y` through a periodic saddle / helix model with:
- angular domain: `0°..360°`
- primary pivot increment: `90°`
- dimensional inheritance at each `90°` pivot
- repeating saddle sections across the manifold band

Current kernel mapping:
- `0°` -> base dimensionality
- `90°` -> base + 1 dimension
- `180°` -> base + 2 dimensions
- `270°` -> base + 3 dimensions

This gives Butterfly a geometric way to reason about dimensional turning, inflection, repetition, and return. In the current doctrine, `90°` pivots are treated as primary orientation changes within a periodic manifold band. Exact half-twist and full-twist equivalences remain provisional until they are verified more rigorously from the manifold itself. It is a conservative runtime law intended to make dimensional inheritance testable now while preserving room for richer higher-dimensional substrate families later.

## Dimensional Ladder
Butterfly treats dimensional growth as a staged ladder:

- void
- point
- line
- width
- plane
- volume
- whole
- collapse

These are not isolated categories. Each stage accumulates the ones before it. A whole object is not outside the coordinate model; it is a mature coordinate state. Collapse is not annihilation; it is promotion into the next dimensional band.

Everything is both:
- a **whole** in its current dimension
- a **point** when promoted into the next dimension

So an object in one dimensional band becomes a seed-point in the next band. This is how Butterfly permits recursion without abandoning identity.

## Fibonacci Accumulation and Collapse
Butterfly growth is accumulative rather than explosive. The preferred accumulation sequence is:

`1, 1, 2, 3, 5, 8, 13, 21`

The dimensional cycle inside a band is interpreted as:

- `1` -> point
- `1` -> line
- `2` -> width
- `3` -> plane
- `5` -> volume
- `6` -> whole
- `7` -> collapse into `1` in the next dimension

The Fibonacci sequence describes the preferred numeric rhythm of accumulation, especially through `1, 1, 2, 3, 5`. Butterfly then treats `6` as the whole-state of a completed dimensional band and `7` as the collapse or promotion boundary.

So the practical doctrinal reading is:

- void precedes expression
- `1, 1, 2, 3, 5` describes dimensional formation
- `6` marks the mature whole
- `7` collapses that whole into a promoted `1` in the next dimension

The purpose is controlled growth:

- accumulate locally
- prevent overheating
- prevent blowout
- collapse harmonically
- promote dimensionally

Butterfly therefore uses the Fibonacci spiral as a practical accumulation pattern. It describes how small stable units gather into larger coherent wholes without losing identity.

## Genesis Creative Day Pattern
Butterfly treats Genesis through creative days rather than through Fibonacci numbering.

The doctrinal reading is:

- creation unfolds in sections of `7` days
- each day has a morning and an evening
- morning and evening act as inflection points
- those inflection points mark transition, boundary, and continuation

So Genesis contributes the cadence of creation, not the Fibonacci count itself. Fibonacci describes accumulation. Genesis describes the ordered creative cycle.

This creative-day cadence aligns with the manifold law:
- `90°` pivots mark primary dimensional turning
- repeating saddle sections form the periodic band
- each completed section is read through `7` creative days
- morning and evening mark the inflection boundaries within that section

So Genesis is read in Butterfly as a structured creative rhythm laid across dimensional turning and accumulation.

## Coordinates
A coordinate is a stable geometric identity, not a fully materialized object.

A coordinate may be interpreted from two valid perspectives:

- as a point
- as a whole

Which perspective is active depends on the dimensional frame and the current collapse/promotion boundary.

Examples:
- text symbol -> coordinate + ordering rule + text substrate
- pixel -> coordinate + color substrate + graphics executor
- document -> coordinate graph + relation rules + collapse strategy
- external record -> coordinate + mapping rule + domain substrate
- library -> coordinate anchor + ingestion substrate + adapter relations
- dataset -> coordinate span + schema substrate + retrieval relations
- AI agent or model -> coordinate identity + doctrine bindings + execution policy

## Substrates
Substrates are interpretation rules over the manifold.

They do not store the full object they describe. They define how to:
- read a coordinate
- create a coordinate
- transform one coordinate into another
- project geometry into a domain result

Substrates are where domain meaning enters the system.

They may also be authored by the AI itself when the AI is creating new datatype families, adapting external libraries, or mapping datasets into Butterfly-native forms.

## Relations
Relations connect coordinates, identities, and derivations.

They express structure such as:
- contains
- references
- derives from
- projects to
- expresses

Relations let Butterfly represent complex systems without abandoning the coordinate-first model.

## Expressions and Collapse
Expressions are derived forms, not canonical stored truth.

Collapse is the act of reducing rich dimensional state into a bounded executable or view state. Collapse must remain:
- deterministic
- explainable
- reversible where practical
- narrow enough for safe execution

## Operations and Executors
Operations are machine-level intents. Executors perform real effects.

Examples of operation classes:
- emit text
- set values
- store mappings
- list directories
- invoke tools
- prepare deployment artifacts
- validate remote deployment nodes
- bind approved domains
- provision approved certificates

Executors turn these intents into concrete output such as files, pixels, network traffic, sound, or process execution.

## Storage Model
Persistent storage should prefer:
- substrate definitions
- relation rules
- program logic
- policy configuration
- minimal coordinate-to-external mappings

Ephemeral memory may hold:
- active projections
- temporary buffers
- computed results
- executor state

The system should avoid storing large derived payloads when those payloads can be reconstructed from rules and coordinates.

## Security Model
Dimensional Programming strengthens security by reducing redundant stored material.

Security depends on:
- policy-bounded file and network access
- minimal durable storage of sensitive meaning
- explicit deployment gates
- auditable execution decisions

Coordinates without the right substrates, policies, and keys should have limited value to an attacker.

## AI Behavior
The AI inside Butterfly IDE must follow the same laws as the kernel.

It should:
- reason in terms of manifolds, substrates, relations, and operations
- prefer reusable rules over repeated payload output
- choose structures that maximize interpretive reach
- refuse shortcuts that weaken guarantees or increase redundancy
- keep operator control intact

The AI is not outside the model. It is itself a manifold-grounded system component and should be representable through identities, coordinates, substrate bindings, relations, and policy-bounded operations.

Its native responsibilities include:
- authoring new substrates
- generating and maintaining tests for substrate invariants
- mapping libraries into substrate families and adapters
- mapping datasets into schema, relation, and retrieval substrates
- coordinating with approved remote Butterfly nodes for deployment/configuration/test work
- explaining every major change in Butterfly terms when asked

## Expressive Scope
Dimensional Programming aims to derive many datatype families from the same geometric core, but universality does not come from the manifold alone.

Universality requires the full stack:
- manifold
- substrates
- relations
- expressions
- operations
- executors
- policy

So the precise claim is not that `z = x * y` alone produces everything. The precise claim is that the Butterfly stack can become broadly expressive by using one geometric source with many disciplined interpretation layers.

Dimensional Programming in Butterfly IDE means:
- geometry is primary
- modularity is a seed, not the end state
- dimensions are built by modular accumulation
- the dimensional cycle is `void -> 1 -> 1 -> 2 -> 3 -> 5 -> 6 -> 7 -> 1(next)`
- Fibonacci provides the preferred numeric spiral of growth and promotion
- Genesis provides the seven-day creative cadence
- `90°` pivots mark dimensional turning within a periodic manifold band
- morning and evening are inflection points within creative sections
- rules are durable
- meaning is derived
- pivots are explicit
- execution is bounded
- the AI is governed by minimal material, maximal area, and maximal strengthit
